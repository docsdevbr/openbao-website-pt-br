# O que é o OpenBao?

O OpenBao é um sistema de gerenciamento de segredos e criptografia baseado em
identidade.
Um _segredo_ é qualquer coisa cujo acesso você queira controlar rigorosamente,
como chaves de criptografia de API, senhas e certificados.
O OpenBao fornece serviços de criptografia protegidos por métodos de
autenticação e autorização.
Usando a interface da pessoa usuária, a CLI ou a API HTTP do OpenBao, o acesso a
segredos e outros dados confidenciais pode ser armazenado e gerenciado com
segurança, rigorosamente controlado (restrito) e auditável.

Um sistema moderno requer acesso a uma infinidade de segredos, incluindo
credenciais de banco de dados, chaves de API para serviços externos, credenciais
para comunicação em arquitetura orientada a serviços, etc.
Pode ser difícil entender quem está acessando quais segredos, especialmente
porque isso pode ser específico da plataforma.
Adicionar recursos como rotação de chaves, armazenamento seguro e registros de
auditoria detalhados é praticamente impossível sem uma solução personalizada.
É aí que o OpenBao entra em cena.

O OpenBao valida e autoriza clientes (pessoas usuárias, máquinas, aplicações)
antes de conceder acesso a segredos ou dados confidenciais armazenados.

## Como o OpenBao funciona?

O OpenBao funciona principalmente com tokens, e um token é associado à política
do cliente.
Cada política é baseada em caminhos, e as regras da política restringem as ações
e a acessibilidade aos caminhos para cada cliente.
Com o OpenBao, você pode criar tokens manualmente e atribuí-los aos seus
clientes, ou os clientes podem fazer login e obter um token.

O fluxo de trabalho principal do OpenBao consiste em quatro etapas:

* **Autenticação:** A autenticação no OpenBao é o processo pelo qual um cliente
  fornece informações que o OpenBao usa para determinar se ele é quem diz ser.
  Assim que o cliente é autenticado com um método de autenticação, um token é
  gerado e associado a uma política.
* **Validação:** O OpenBao valida o cliente em relação a fontes confiáveis de
  terceiros, como GitHub, LDAP, AppRole e outras.
* **Autorização**: Um cliente é comparado com a política de segurança do
  OpenBao.
  Esta política é um conjunto de regras que define a quais endpoints da API um
  cliente tem acesso com seu token OpenBao.
  As políticas fornecem uma maneira declarativa de conceder ou proibir o acesso
  a determinados caminhos e operações no OpenBao.
* **Acesso**: O OpenBao concede acesso a segredos, chaves e recursos de
  criptografia emitindo um token com base em políticas associadas à identidade
  do cliente.
  O cliente pode então usar seu token OpenBao para operações futuras.

## Por que usar o OpenBao?

Atualmente, a maioria das empresas possui credenciais espalhadas por toda a
organização.
Senhas, chaves de API e credenciais são armazenadas em texto simples,
código-fonte de aplicações, arquivos de configuração e outros locais.
Como essas credenciais estão por toda parte, essa dispersão pode dificultar e
tornar complexo saber quem tem acesso e autorização para quê.
Ter credenciais em texto simples também aumenta o potencial para ataques
maliciosos, tanto internos quanto externos.

O OpenBao foi projetado pensando nesses desafios.
Ele centraliza todas essas credenciais, definindo-as em um único local, o que
reduz a exposição indesejada delas.
Mas o OpenBao vai além, garantindo que pessoas usuárias, aplicações e sistemas
sejam autenticados e explicitamente autorizados a acessar recursos, além de
fornecer um registro de auditoria que captura e preserva o histórico das ações
dos clientes.

Os principais recursos do OpenBao são:

- **Armazenamento seguro de segredos**: Segredos arbitrários de chave/valor
  podem ser armazenados no OpenBao.
  O OpenBao criptografa esses segredos antes de gravá-los no armazenamento
  persistente, portanto, obter acesso ao armazenamento bruto não é suficiente
  para acessar seus segredos.

- **Segredos dinâmicos**: O OpenBao pode gerar segredos sob demanda para alguns
  sistemas, como Kubernetes ou bancos de dados SQL.
  Por exemplo, quando uma aplicação precisa acessar um cluster Kubernetes, ela
  solicita credenciais ao OpenBao, e o OpenBao gera um token de conta de serviço
  do Kubernetes com permissões válidas sob demanda.
  Após criar esses segredos dinâmicos, o OpenBao também os revogará
  automaticamente após o término do período de concessão.

- **Criptografia de dados**: O OpenBao pode criptografar e descriptografar dados
  sem armazená-los.
  Isso permite que as equipes de segurança definam parâmetros de criptografia e
  que as pessoas desenvolvedoras armazenem dados criptografados em um local como
  um banco de dados SQL, sem precisar criar seus próprios métodos de
  criptografia.

- **Locação e renovação**: Todos os segredos no OpenBao têm uma _concessão_
  associada.
  Ao término da concessão, o OpenBao revogará automaticamente esse segredo.
  Os clientes podem renovar as concessões por meio das APIs de renovação
  integradas.

- **Revogação**: O OpenBao oferece suporte integrado para revogação de segredos.
  O OpenBao pode revogar não apenas segredos individuais, mas também uma árvore
  de segredos, por exemplo, todos os segredos lidos por uma pessoa usuária
  específica ou todos os segredos de um determinado tipo.
  A revogação auxilia na troca de chaves, bem como no bloqueio de sistemas em
  caso de intrusão.
